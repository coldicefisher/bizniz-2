# Error classification in the test loop

When pytest exits with a collection error (codes 2 or 4), the orchestrator
must decide whether the bug is in the **test** file or in our **source**
code. Misclassification wastes iterations regenerating tests that won't
help, or repairing source that wasn't broken.

This doc covers two related guards in
`bizniz/orchestrator/coding_orchestrator.py`:

- `_is_source_import_error` — collection-error router
- `_is_config_file` + writable-config integration — repair-loop file scope

## Collection-error routing

The orchestrator's main loop intercepts pytest exit codes 2 and 4 and
routes through three signals before deciding:

```
 ┌──────────────────────┐
 │ pytest collection    │
 │ failed (exit 2/4)    │
 └──────┬───────────────┘
        ▼
 ┌──────────────────────┐
 │ _is_source_import_   │
 │ error(detail, files) │
 └──────┬───────────────┘
        │
   yes  │  no
   ┌────┴────┐
   ▼         ▼
 repair    regenerate
 source    tests
 code
```

`_is_source_import_error` returns True if **any** of three signals fires:

1. **Traceback frame match.** The error text contains a frame located
   inside one of our workspace source files. Recognised patterns:
   - `<filepath>:<line>:` markers
   - `File "<filepath>", line <line>`

   A frame in `tests/...` does NOT count — that's a test problem.

2. **Failing-import module match.** The first `> from X import` line in
   the error refers to one of our source modules. Catches the classic
   "test imports our broken module" case.

3. **Source-file path mention** under `ImportError` or `ModuleNotFoundError`.
   Cheapest fallback for cases where the test couldn't even start importing.

Negative on all three → False → orchestrator regenerates the test.

### Cases verified

| Failure pattern | Source side? | Why |
|---|---|---|
| `pet_groomer/app.py:7: NameError: FastAPI` | YES | traceback frame in source |
| `File "/workspace/pet_groomer/app.py", line 7` | YES | quoted-path frame in source |
| `> from pet_groomer.routers.X import …` (ImportError) | YES | failing import is our module |
| `ModuleNotFoundError: pet_groomer.repositories.datastore` + path mention | YES | path in ImportError context |
| `fixture 'foo' not found` (line in `tests/`) | NO | fixture problem in test |
| Test imports `some_third_party_lib` not installed | NO | not our module |
| Frame only in `/workspace/tests/test_routes.py` | NO | test problem, not source |
| Empty / null `error_detail` | NO | nothing to classify |

See `bizniz/orchestrator/tests/test_collection_error_classification.py`
for the test cases.

## Writable-file scope during repair

When a repair iteration runs, the autocoder is shown two file lists:

- **Writable**: source + test files for the current issue
- **Read-only**: prior-issue source files needed for context

If the AI tries to edit a read-only file, the orchestrator strips the
change and warns the LLM next iteration. This is correct for cross-issue
contamination (e.g. issue 2 shouldn't rewrite issue 1's models).

But it bit us when the actual root cause was in **universal project
config** that no issue declares as a target file:

- Frontend Layer 0 stalled because the React skeleton's `package.json`
  needed `ts-jest` added — but `package.json` wasn't in any issue's
  `target_files`, so all repair attempts got filtered.
- Backend Layer 2 spent a long pro repair editing `pyproject.toml`
  for a pytest config issue, all changes filtered.

### Fix: config-file allowlist

`_is_config_file(path)` is True when the path's basename is in
`_CONFIG_FILENAMES` — a hand-maintained set of universal project config:

| Category | Filenames |
|---|---|
| Python | `pyproject.toml`, `setup.cfg`, `setup.py`, `pytest.ini`, `tox.ini`, `conftest.py`, `requirements*.txt` |
| JS/TS | `package.json`, `tsconfig*.json`, `jest.config.*`, `vitest.config.*`, `vite.config.*`, `webpack.config.js`, `angular.json`, `babel.config.js` |
| Containerization | `Dockerfile`, `docker-compose.yml`, `docker-compose.yaml` |

At repair time, the orchestrator scans `workspace.list_relative_files()`,
auto-loads any matching config files, and adds them to `writable_paths`
**and** `relevant_files`. The autocoder sees them as writable; the
post-repair filter permits them.

Lockfiles (`package-lock.json`, `yarn.lock`, `Cargo.lock`) are
deliberately excluded — those are generated artifacts that should be
regenerated by re-running the package manager, not hand-edited by the AI.

### Cases verified

| File | Allowed for repair? |
|---|---|
| `package.json` | YES |
| `frontend/package.json` (any depth) | YES |
| `infra/development/backend/Dockerfile` | YES |
| `pyproject.toml`, `pytest.ini`, `requirements.txt` | YES |
| `app/main.py`, `src/App.tsx`, `tests/test_models.py` | NO (regular code) |
| `package-lock.json`, `yarn.lock`, `Cargo.lock` | NO (lockfiles) |

See `bizniz/orchestrator/tests/test_collection_error_classification.py::test_config_allowlist_*`
for the test coverage.
