# 2026-05-01 — Build vs Evolve: two-mode pipeline strategy

Strategic direction for how bizniz handles large/long-lived apps,
captured during the V5 verification run while skeleton-contract
fixes were proving out.

## The core decision

**bizniz operates in two modes**, with a clean seam between them:

1. **Build mode** — the greenfield path we ship today
   (`architect.build()`). Spec-driven decompose → engineer per
   service → integration phase. Works for new apps with up to
   ~30-50 services.

2. **Evolve mode** — adding features to an already-built app
   (`architect.evolve()` exists as a stub, not built yet). Reads v0
   artifacts as anchors, uses discovery tools (grep / read /
   find-references) to navigate existing code, dispatches engineers
   to ADD without rewriting.

We ship build mode now. We add evolve mode when a real customer app
needs its second feature — not before.

## Why both modes are needed

The discoverer → business_manager → marketer → builder pipeline
produces apps that customers iterate on. If bizniz can only do
greenfield, customers are stuck after v0 — they either accept the
v0 forever or hand-edit code. Neither is acceptable.

The build/evolve seam means bizniz can hand off a v0 **and** keep
adding features against it later via the same agent machinery.

## The four artifacts that bridge the modes

Every successful build run produces these. Together, they're what
evolve-mode agents read FIRST when starting work on an existing
app:

1. **Architecture digest JSON** — `<project>/docs/runs/<job_id>.json`
   captures services, frameworks, ports, dependencies. Already
   shipping today.

2. **Integration tests against the live stack** — the
   `tests/integration/test_*_api.py` files generated by
   `HTTPApiTester` (shipped 2026-05-01). These are the **executable
   spec** — they pin domain behavior and are run as a baseline
   before evolve agents touch anything. Already shipping today.

3. **`SKELETON.md` per service** — directory contract. Tells
   future agents what's safe to edit and what's read-only. Shipped
   2026-05-01 across all 5 skeletons (fastapi, react, angular,
   teams, saas) plus the no-skel fallback templates.

4. **Captured OpenAPI / contract files** — `<project>/contracts/
   <svc>.openapi.json`. Lets evolve-mode agents know exactly what
   endpoints exist and what shapes they accept. Already shipping
   today (`bizniz.integration.contracts`).

## Why this works (and what makes it load-bearing)

- **Greenfield is easier for spec-driven** because there's no prior
  code to clash with. Spec-driven hits a wall on existing
  codebases — you can't fit them in a prompt. So we use specs for
  what they're good at and discovery for what specs can't reach.
- **Integration tests pin behavior** — they're the durable contract,
  not decoration. A v1 agent that breaks v0's integration tests
  knows immediately. This is why the integration phase we shipped
  2026-05-01 isn't optional — it's the foundation of evolve mode.
- **Discovery tools** (grep workspace, find references, list dir)
  fill the gap between "what the digest claims" and "what the code
  does today." Same pattern Claude Code itself uses internally —
  read artifacts, fall back to grep.

## The strategy applied to specific scenarios

### Sockets / realtime apps

Route to the **saas skeleton** — it already ships the realtime
backbone (WebSocket server, Redis pub/sub, store-consumer with
processing locks). v0 builds with the saas skeleton produce a
working realtime stack with integration tests covering WebSocket
flows. Adding feature X later is then "evolve agent reads
architecture.json, lists existing channels, runs the realtime
integration tests as baseline, then writes new code that doesn't
break those tests."

The integration tests for WebSocket flows are **non-negotiable**
in v0 — they're the contract evolve mode will trust.

### Customer-facing apps

Same pattern: build v0 with full integration tests covering the
customer's stated use cases. When the customer says "add feature
X," that's the trigger for evolve mode (with discovery tooling).
Not before.

### Internal tools / one-shot apps

Build mode is sufficient. Evolve mode is over-engineering for
apps that ship and don't change.

## What this means for what we build right now

- **Don't build evolve-mode discovery tooling yet.** Premature
  until we have a real evolving app.
- **DO ensure build-mode outputs the four artifacts cleanly.** The
  artifacts are stale-able — if we forget to capture one, evolve
  mode is stuck.
- **Integration test quality is load-bearing.** Bad v0 tests =
  misleading v1 agents. The integration phase shipped today is
  what makes the whole evolve-mode plan viable.
- **When a customer asks for feature X on an existing app**, that's
  the trigger to build evolve mode — not the trigger to add
  another knob to build mode.

## Roadmap (rough)

| Phase | What | When |
|---|---|---|
| Now | Build mode reliable + four artifacts captured per run | Shipped 2026-05-01 |
| Soon | First customer app built with discoverer→builder pipeline | Next |
| Trigger | Customer requests feature X on existing app | Reactive — when it happens |
| Then | Build `architect.evolve()` with grep/read/find-refs tools, anchored on v0 artifacts | After trigger |
| Later | Iterate on evolve mode based on what real maintenance feels like | Ongoing |
